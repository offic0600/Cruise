# Forge 最小系统轮廓

> 日期: 2026-03-12
> 关联文档: `forge-mvp-scope.md`、`forge-user-flow.md`

## 一、文档目标

本文档定义 Forge v1 的最小系统结构，用于约束架构设计，避免在 MVP 阶段引入过多平台层组件。

## 二、系统目标

Forge v1 的系统目标只有一个：

**支撑单任务交付闭环稳定运行。**

因此，系统结构应围绕“输入任务、装配上下文、执行任务、展示结果、确认结果”五件事展开。

## 三、最小系统结构

```text
Web IDE
-> Session API
-> Task Orchestrator
-> Context Service
-> SuperAgent Runtime
-> Workspace Adapter / Tool Adapter
-> Result Store
```

## 四、模块说明

### 4.1 Web IDE

职责：

- 展示文件树和编辑区
- 接收用户任务输入
- 展示执行过程
- 展示结果与待确认项

不承担：

- 复杂业务编排
- 长期规则治理

### 4.2 Session API

职责：

- 接收前端任务请求
- 创建和维护任务会话
- 向前端返回任务状态
- 对接实时消息流

这是 v1 的入口层，不需要承载复杂领域逻辑。

### 4.3 Task Orchestrator

职责：

- 驱动单次任务流程
- 控制任务状态流转
- 决定何时装配上下文
- 决定何时调用 Agent
- 决定何时进入结果确认

这是 v1 的核心业务中枢。

### 4.4 Context Service

职责：

- 按任务选择相关文件
- 读取项目规则文件
- 收集最小必要上下文

限制：

- v1 不做复杂跨系统知识编排
- v1 不做大规模长期记忆体系

### 4.5 SuperAgent Runtime

职责：

- 生成计划
- 调用工具
- 执行任务
- 输出结果总结

原则：

- 保持单 Agent 结构
- 允许轻量角色切换
- 不引入多 Agent 协同框架

### 4.6 Workspace Adapter / Tool Adapter

职责：

- 文件读取与写入
- Git 状态读取
- 命令执行
- 基础知识检索

这一层是后续扩展 MCP 的自然位置，但 v1 只接入最小必要能力。

### 4.7 Result Store

职责：

- 保存任务输入
- 保存执行摘要
- 保存修改文件列表
- 保存确认结果

v1 不需要复杂分析型数据仓设计，只需要满足任务追溯和基础回看。

## 五、最小状态流转

任务状态建议收敛为：

1. `draft`
2. `understanding`
3. `ready`
4. `running`
5. `review`
6. `done`
7. `failed`

说明：

- `draft`：用户刚输入任务
- `understanding`：系统正在理解任务与装配上下文
- `ready`：计划已生成，等待用户确认执行
- `running`：Agent 正在执行
- `review`：结果已生成，等待用户确认
- `done`：用户接受结果
- `failed`：执行失败或用户放弃

## 六、v1 不引入的系统层

以下系统层不应在 v1 强行落地：

- 多 Agent 调度层
- 企业级权限治理中心
- 通用规则引擎平台
- 完整 MCP 注册中心
- 复杂观测分析平台
- 平台级 KPI 大盘

## 七、后续演进接口

虽然 v1 要收敛，但以下边界应预留扩展点：

### 7.1 Context Service

后续可接入：

- 团队知识
- 项目模板
- 共享记忆

### 7.2 Tool Adapter

后续可接入：

- 数据库能力
- 服务图谱
- 制品系统
- 可观测性系统

### 7.3 Result Store

后续可扩展：

- 团队级任务分析
- 任务模式萃取
- 交付效果评估

## 八、架构判断标准

如果某个新模块不能直接回答以下问题之一，默认不应进入 v1：

1. 它是否让任务闭环更稳
2. 它是否让上下文装配更准
3. 它是否让执行过程更透明
4. 它是否让结果确认更清晰
