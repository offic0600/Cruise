# Forge 用户流程

> 日期: 2026-03-12
> 关联文档: `forge-mvp-scope.md`

## 一、文档目标

本文档定义 Forge v1 的单任务用户闭环，确保产品设计、交互设计和系统结构围绕同一条主流程展开。

## 二、核心流程

```text
进入工作台
-> 输入任务
-> 系统理解任务
-> 装配上下文
-> Agent 生成计划
-> 用户确认开始执行
-> Agent 执行任务
-> 系统展示结果
-> 用户确认或继续调整
-> 沉淀记录
```

## 三、详细流程

### 3.1 进入工作台

用户进入 Forge Web IDE，看到当前仓库、文件树、主编辑区和任务输入区。

目标：

- 明确当前操作对象是哪个仓库
- 明确当前工作上下文是否可用

### 3.2 输入任务

用户输入任务，例如：

- 修改一个接口
- 修复一个 bug
- 补充一份文档
- 调整一个配置项

系统应支持用户附加补充信息：

- 目标文件
- 预期结果
- 约束条件

### 3.3 系统理解任务

系统基于用户输入生成任务理解结果，至少包括：

- 任务目标
- 可能涉及的文件
- 风险提示
- 预估执行范围

此阶段的重点不是立即动手，而是减少理解偏差。

### 3.4 装配上下文

系统围绕任务收集最小必要上下文：

- 当前仓库相关文件
- 项目级规则文件
- 最近执行记录
- 必要的外部知识或工具能力

这一阶段的目标是控制上下文质量，而不是盲目加载更多信息。

### 3.5 Agent 生成计划

Agent 输出一个可读、可审阅的执行计划，通常包括：

1. 要查看哪些文件
2. 要修改哪些内容
3. 要执行哪些验证动作
4. 预期产出是什么

### 3.6 用户确认开始执行

用户确认系统对任务的理解和执行计划没有明显偏差后，进入执行阶段。

如用户发现偏差，应支持继续补充说明，而不是直接跳过。

### 3.7 Agent 执行任务

执行阶段需要透明展示以下信息：

- 当前步骤
- 读写了哪些文件
- 执行了哪些命令
- 是否出现错误
- 是否需要用户补充信息

### 3.8 系统展示结果

执行完成后，系统应展示：

- 修改文件列表
- 关键变更摘要
- 验证结果
- 风险与待确认项

### 3.9 用户确认或继续调整

用户有三种典型动作：

1. 接受结果
2. 提出补充修改
3. 放弃本次结果

这一步决定 Forge 是否能形成稳定的人机协作闭环。

### 3.10 沉淀记录

任务结束后，系统应保留最小必要记录：

- 原始任务输入
- 执行摘要
- 修改文件清单
- 最终确认结果

这类沉淀将成为后续 v2 团队化能力的基础。

## 四、关键交互原则

### 4.1 先理解，再执行

避免用户刚输入任务系统就直接开始大范围改动。

### 4.2 过程透明

用户必须看得见系统读了什么、做了什么、为什么这样做。

### 4.3 结果可确认

任何任务完成后都必须进入“结果确认”状态，而不是默认视为完成。

### 4.4 支持多轮补充

用户应能在同一任务上下文中继续追问、修正、细化，而不是每次从零开始。

## 五、流程外不做的事

v1 不把以下动作放入主流程：

- 多人协作审批链
- 跨项目任务编排
- 自动发布上线
- 平台级经营分析

## 六、成功标准

若该流程可稳定支持 80% 以上的单仓库日常任务，Forge v1 的主流程定义可视为成立。
