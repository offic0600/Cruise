# Cruise 设计对比汇报

> 对比对象：
> `docs/baselines/design-baseline.md` 中的原始设计基线
> 与当前代码实现状态
>
> 目标：
> 从“原设计不足 / 当前优势 / 业务价值”三个维度，说明当前实现相较原始设计的演进方向与实际收益。

---

## 一、原设计不足

原始设计基线完成了项目管理系统的基础骨架，但整体仍以传统模块拆分为主，存在明显的结构性局限。

### 1. 领域模型割裂，跨对象能力难统一

原始设计以 `Requirement`、`Task`、`Defect` 为三套核心对象分别建模，并分别配套控制器、服务和数据表。这种设计在 Phase 1 交付期是合理的，但它更适合“先把基础 CRUD 做出来”，不适合后续平台化演进。

其主要问题包括：

- 同类能力需要重复建设，例如列表、详情、筛选、状态流转、工时、搜索、标签、活动记录都容易在多个对象上重复实现。
- 对象之间虽然存在业务联系，但系统层面缺少统一抽象，难以自然表达“需求拆任务、任务挂缺陷、缺陷回溯任务”这类链路。
- 当业务对象继续扩展时，系统很容易继续沿用“新增一个对象，就新增一套模块”的方式，维护成本会快速上升。

### 2. 标签体系过窄，只服务单一对象

原始设计中的标签能力主要体现为 `requirement_tag`，标签范围局限在需求对象上。这意味着标签并不是一个面向工作空间的通用分类能力，而只是局部字段设计。

这种方案的问题在于：

- 标签无法统一覆盖任务、缺陷等其他工作对象。
- 标签无法形成跨团队、跨项目的一致语义体系。
- 标签不能承担筛选、聚合、视图配置、知识沉淀等更高层次的组织作用。

### 3. 缺少“视图”这一工作组织层

原始设计中已有 dashboard 和 analytics，但它们更偏向固定报表与统计接口，不等同于用户工作台所需要的“视图能力”。

缺失视图层会带来几个问题：

- 用户只能进入固定页面或模块工作，无法按自己的角色、职责、关注范围组织数据。
- 系统难以沉淀“我的待办”“高优先级问题”“本周待审”“某项目风险项”这类可复用工作入口。
- 数据是存在的，但用户与数据之间缺少稳定的操作界面，实际效率会低于系统能力上限。

### 4. 搜索与筛选偏模块化，难形成统一入口

原始设计更多是各模块各自提供列表接口与查询能力，缺少跨实体、跨上下文的统一检索入口。

这会导致：

- 关键词检索停留在单模块范围内，无法覆盖事项、项目、文档等多种对象。
- 搜索结果不能自然衔接到工作上下文，用户需要在不同模块间来回跳转。
- 结构化筛选难以和关键词搜索结合，查询体验容易碎片化。

### 5. 更像“系统集合”，而不是“协同平台”

从原始基线看，系统重点在于需求、任务、团队成员、缺陷、报表、集成和 Agent 能力的逐步叠加。这种做法适合阶段性交付，但整体上仍是“功能模块的集合”，而不是围绕统一工作对象、统一界面和统一协作流程构建的平台。

因此，原始设计的核心不足，不在于功能少，而在于：

- 抽象层次偏低
- 组织方式偏模块化
- 协作体验缺少统一载体

---

## 二、当前优势

当前实现相较原始设计，最大的变化不是“功能数量增加”，而是形成了更稳定的平台型结构。系统已经从模块拼接，演进为围绕统一工作对象、统一视图入口和统一组织体系展开的工作台。

### 1. 统一 Issue 模型，显著降低系统割裂

当前实现已经以 `Issue` 作为统一事项模型，使用 `type`、`state`、`priority`、`parentIssueId`、`labels`、`customFields` 等字段表达不同业务类型和层级关系。

这种设计的优势在于：

- 需求、任务、缺陷等对象可在同一模型中表达，减少重复建模与重复接口建设。
- 父子关系通过 `parentIssueId` 统一承载，能够更自然地表达工作拆解链路。
- 标签、自定义字段、活动记录、附件、评论、关系等通用能力可以一次建设，多处复用。
- 新增对象类型时，不需要复制整套模块，扩展成本更低。

这意味着当前设计已经从“对象驱动的页面系统”，升级为“统一事项驱动的平台架构”。

### 2. 保存视图成为正式能力，用户拥有可复用工作台

当前系统已经引入独立的 `View` 模型，支持保存过滤条件、分组方式、排序规则、可见范围与系统视图标记，并在前端形成了实际可用的视图页面。

这比原始固定报表设计更强，原因在于：

- 视图不再只是展示结果，而是用户组织工作的方式。
- 用户可以围绕角色、项目、优先级、状态等维度形成稳定入口。
- 系统能够沉淀标准视图，也允许用户形成个人或团队工作习惯。

视图能力的落地，使系统开始具备“工作操作系统”的特征，而不只是“数据录入与查询系统”。

### 3. 标签系统升级为分层标签系统

当前标签能力已经从单一 `requirement_tag` 升级为可复用的标签定义与绑定体系，并支持 `WORKSPACE` 与 `TEAM` 两级作用域。

这套设计的优点是：

- 标签不再依附于某个单独对象，而是成为组织数据的横向语义层。
- 团队可以维护局部标签，工作空间可以维护全局标签，既统一又保留灵活性。
- 标签可直接服务筛选、视图、详情编辑、工作流组织和后续统计分析。
- 前端已经支持在详情页内直接选取或新建标签，形成闭环体验。

换句话说，标签已经从“字段补充”升级为“信息架构能力”。

### 4. 关键词搜索与结构化筛选开始融合

当前实现中，搜索不再局限于单一模块，而是已经覆盖 `issues / projects / documents` 三类对象；同时，Issue 查询本身还支持关键字与自定义字段过滤并用。

这带来了两个关键提升：

- 用户可以从统一入口查找不同类型的信息对象，而不是先猜信息属于哪个模块。
- 搜索不只是“模糊找东西”，而是逐步和结构化筛选结合，具备成为统一工作入口的潜力。

这类能力对于平台型产品非常重要，因为它直接影响信息可达性和协作效率。

### 5. 统一模型之上已经叠加协同能力

当前代码实现已经不止于 Issue、Project 两类核心对象，而是扩展到了 Initiative、Roadmap、Workflow、Doc、Comment、Notification、Template、Recurring、Email Intake 等对象与能力。

这说明当前实现的优势不是某一两个页面更丰富，而是已经形成以下特征：

- 事项有上下文
- 事项有流程
- 事项有文档
- 事项有协作痕迹
- 事项有自动化延展空间

也就是说，系统正在从“项目管理工具”过渡为“协同运营平台”。

---

## 三、业务价值

从业务角度看，当前实现相较原始设计的价值，主要不是技术架构更先进，而是它更接近真实团队的工作方式，能支撑更复杂的组织场景与更长期的产品演进。

### 1. 降低认知成本，提升团队协同一致性

统一 Issue 模型之后，不同角色面对的是同一套核心对象和同一套交互逻辑。产品、研发、测试、项目管理不需要在多个相互割裂的模块间切换理解方式。

其直接价值包括：

- 降低培训与上手成本
- 降低跨角色沟通成本
- 降低系统使用过程中的语义偏差

对业务来说，这意味着系统更容易在团队内部形成统一操作语言。

### 2. 把“信息存在”变成“信息可用”

原始设计下，很多数据即使存在，也常常需要依赖用户自己去找、去拼、去记。当前视图、标签、搜索能力结合后，信息开始具备可组织、可沉淀、可复用的特征。

这会直接提升：

- 日常工作检索效率
- 重点事项发现效率
- 团队共享工作方法的能力

从管理角度看，这会让系统从“记录工具”转向“决策与执行工具”。

### 3. 支撑规模化扩展，而不是每次扩展都重做一套

统一模型、通用标签、视图抽象、自定义字段等能力，决定了后续产品迭代可以更多围绕平台能力扩展，而不是继续沿用“新对象 + 新表 + 新页面 + 新接口”的模式。

这带来的业务收益包括：

- 新需求上线更快
- 系统复杂度增长更可控
- 长期维护成本更低
- 新业务场景更容易接入

这对产品中长期演进价值很高，因为它决定了系统能否承载未来更多团队和更多业务形态。

### 4. 更容易形成标准化管理与个性化工作的平衡

原始设计更偏标准模块；当前实现通过工作空间标签、团队标签、保存视图、自定义字段等能力，开始兼顾统一规范与局部灵活性。

这意味着系统既可以：

- 在组织层面建立标准流程、标准字段、标准视图

也可以：

- 允许团队按自身节奏配置分类方式和工作入口

这种平衡能力，正是平台型产品在真实企业环境中能否落地的关键。

### 5. 为智能化与自动化能力提供更坚实载体

原始基线中已经有 SuperAgent + Skill 的设计与实现方向，但如果底层对象模型不统一，Agent 实际能调用和编排的能力也会受到限制。

当前实现由于已经具备统一 Issue 模型、标签体系、文档关联、视图组织与搜索入口，因此后续无论是 Agent 辅助分析、自动分发、自动归类、自动总结还是工作流推荐，都拥有更清晰的数据基础与交互入口。

因此，当前设计的业务价值还体现在：

- 更容易把 AI 能力嵌入真实工作流
- 更容易形成自动化闭环
- 更容易沉淀高质量组织数据

---

## 结论

原始设计解决的是“先把项目管理核心模块搭起来”的问题，属于阶段性合理设计；当前实现解决的则是“如何把这些能力组织成一个真正可协作、可扩展、可沉淀的平台”的问题。

如果用一句话概括这次演进：

**原设计偏模块式项目管理系统，当前实现已经明显演进为以统一 Issue 模型为核心、以视图和标签为组织手段、以搜索和协作为入口的平台化工作台。**

这也是当前版本相较原始基线最核心、最有价值的设计升级。
