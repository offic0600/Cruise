# AI Coding 最佳实践指南

> **来源**：基于 Forge 平台 28 个 Session、29 个 Bug、368 个验收测试用例的完整实践提炼
> **适用**：Cruise 项目全生命周期
> **更新**：每次重大经验沉淀后更新本文档

---

## 核心理念

> **"用文档对抗 AI 遗忘，用验收测试对抗质量腐化，用双基线对抗设计偏移。"**

Claude Code 的上下文窗口是有限的，但项目复杂度是无限增长的。
必须建立**外部记忆系统**来弥补 AI 的遗忘，并用**开发纪律**保证记忆的准确性。

---

## 一、上下文与记忆管理

### 1.1 CLAUDE.md — 项目记忆的核心载体

CLAUDE.md 是每次 Session 自动加载的唯一持久化文件，是 AI 知识的"大脑"。

**应该写入 CLAUDE.md 的内容**：

| 类别 | 内容 | 示例 |
|------|------|------|
| 交互偏好 | 语言、风格、行动方式 | "用中文，行动优先" |
| 技术栈 | 语言版本、框架版本 | "JDK 21（不是17！）" |
| 架构决策 | 关键设计选择 | "SuperAgent 优于 Multi-Agent" |
| 已知陷阱 | 踩过的坑 | "Kotlin 枚举默认大写，需加 @JsonValue" |
| 开发纪律 | 工作流程规则 | "每次 Session 结束更新 Logbook" |
| 构建命令 | 快速启动 | `./gradlew bootRun` |

**演进规律**：
Forge 的 CLAUDE.md 从 67 行增长到 181 行，跨越 29 个 Session 不断沉淀。
**不要一次写完，而是随着项目推进持续补充。**

**CLAUDE.md 四层架构**：
```
企业策略层（最高优先级，全局生效）
  └── 项目层 CLAUDE.md（项目根目录，每次 Session 自动加载）
        └── 模块规则 .claude/rules/（按 glob 匹配，按需加载）
              └── 用户层 ~/.claude/CLAUDE.md（个人偏好，跨项目）
```

**按需加载节省 Token**：
子目录的 CLAUDE.md 只有当 Claude 读取该目录文件时才加载。
可以把模块专属规则放在 `.claude/rules/` 下并配置 glob 匹配：
```markdown
---
globs: ["src/api/**/*.ts"]
---
# API 安全规则
- 所有端点必须验证 JWT
- 输入必须用 Zod 验证
```

### 1.2 上下文窗口管理

**200K Token 窗口分配参考**：

```
系统 Prompt                    5-15K tokens
CLAUDE.md（所有位置）           1-10K tokens
.claude/rules/*.md             变量
对话历史 + 工具调用结果          140-150K tokens（主要空间）
响应生成缓冲区                   40-45K tokens
```

**4 区间管理**：

| 区间 | 使用率 | 操作 |
|------|--------|------|
| 🟢 安全 | 0-50% | 正常工作 |
| 🟡 关注 | 50-70% | 开始注意 Token 消耗 |
| 🟠 干预 | 70-90% | 执行 `/compact` |
| 🔴 危险 | 90%+ | 立即 `/compact` 或重开 Session |

**核心操作**：

- **`/compact`**：压缩对话历史，保留关键信息。可以加指向：`/compact focus on the API changes`
- **`/clear`**：完全清空对话，CLAUDE.md 仍然生效（适合切换任务）
- **`#` 快速记忆**：`# build fails if NODE_ENV isn't set`（立即添加到 Auto Memory）
- **`/memory`**：打开记忆文件选择器手动管理

**留白哲学**：保留 25% 上下文窗口作为工作空间。上下文填满时 LLM 性能下降 20-30%。

### 1.3 Auto Memory（自动记忆系统）

Claude Code 自动将有价值的上下文保存到：
```
~/.claude/projects/<project>/memory/
├── MEMORY.md              # 精简索引（每次 Session 加载前 200 行）
├── debugging.md           # 调试经验
├── api-conventions.md     # API 设计决策
└── ...                    # 其他主题文件
```

**最佳实践**：
- MEMORY.md 保持精简（<200 行），只放索引
- 主动要求 Claude 保存："remember that we use pnpm, not npm"
- 每个任务用全新上下文启动，状态通过文件和 git 持久化

## 二、Session 工作流

### 2.1 Session = 最小工作单元

每个 Session 遵循固定结构（微型 PDCA 循环）：

```
目标声明 → 实施（代码/文档）→ Bug 修复 → 文件变更表 → 经验沉淀 → 统计快照 → Git 提交
```

**Session 开始**：
1. 回顾 Logbook 最后几条，恢复上下文
2. 声明本次 Session 的单一目标（一句话）
3. 如果是大型任务，先进入 Plan Mode

**Session 结束**：
1. 更新 Logbook（文件变更表 + Bug + 经验 + 统计）
2. 交叉校验基线（如果 Phase 结束）
3. Git 提交（语义化 commit）

### 2.2 日常工作流参考

```bash
# 1. Session 开始：回顾上下文
"读一下 dev-logbook.md 最后一个 Session，告诉我当前状态"

# 2. 定义今日目标
"今天我们要完成：[具体目标]"

# 3. 大型任务先规划（Plan Mode）
# 用 /plan 进入，Claude 提交计划后你审批再执行

# 4. 工作中监控 Token
/cost   # 检查用量，>50% 考虑 /compact

# 5. 切换重大任务时清空上下文
/clear

# 6. Session 结束
"更新 Logbook，提交 git"
```

## 三、规划与 Phase 管理

### 3.1 Phase 迭代模型

**核心原则**：每个 Phase 有明确的验收标准，Phase 结束时做验收测试 + 基线更新。

**Phase 命名规范**：
- 格式：`Phase X.Y — 语义标题`（如 `Phase 1.6 — AI 交付闭环`）
- 新增中间阶段用小数点扩展
- **禁止重命名已有编号**（Forge Session 13 做了 69 处重命名，耗费了整个 Session）

**Phase 节奏参考**（Forge 实证）：

| 阶段 | 占比 | 说明 |
|------|------|------|
| 编码实现 | ~60% | 功能开发 |
| 验收测试 + Bug 修复 | ~33% | "让它可信赖"的成本 |
| 文档治理 | ~7% | 基线更新、Logbook |

> **洞察**："写完代码"只是交付的一半，"让它可信赖"才是另一半。

### 3.2 Plan Mode 使用时机

以下情况**必须**先进 Plan Mode：
- 涉及 5+ 文件修改的任务
- 新 Phase 的架构设计
- 重大重构

Plan Mode 防止 Claude 直接行动带来的方向错误，通常节省 30-50% 的返工时间。

## 四、并行 Agent 策略

### 4.1 何时使用并行 Agent

| 场景 | 策略 |
|------|------|
| 初始化项目（10+ 文件） | 拆分为前端/后端 2-3 个并行 Agent |
| 大型功能集（5+ 文件） | 按模块拆分，并行创建 |
| 独立场景的验收测试 | 不同场景并行执行 |
| 文档批量更新 | Logbook + Baseline + 验收测试并行修改 |

### 4.2 并行 Agent 注意事项

- Agent 拥有**独立的上下文窗口**，主会话不被污染
- 子 Agent 完成后只返回摘要（避免大量中间结果占满主上下文）
- 并行 Agent 可能遇到 API rate limit，需要合理安排任务优先级
- Explore Agent 的评估可能误判，关键结论需要手动 double-check

**Forge 实证**：Session 1 同时启动 6 个 Agent，一次性创建 227 文件 / 37,179 行代码。但后续 Session 回归单线程，导致效率未充分释放。**要主动利用并行 Agent。**

## 五、开发纪律四大支柱

### 纪律 1：Logbook 维护（对抗遗忘）

**为什么重要**：Forge 18 个 Session 横跨 3-4 次 context 重置，但项目从未中断，核心原因就是 Logbook。

**每条 Session 记录必须包含**：

```markdown
## Session N — YYYY-MM-DD：[一句话目标]

### 实施内容
| 操作 | 文件 | 说明 |
|------|------|------|
| 新增 | src/xxx.ts | 实现了... |
| 修改 | src/yyy.ts | 修复了... |

### Bug 记录
- **BUG-N**：[描述] → 根因：[...] → 修复：[...]

### 经验沉淀
- [可编码为 CLAUDE.md 已知陷阱的教训]

### 统计快照
| 指标 | 值 |
|------|-----|
| 文件数 | X |
| 测试数 | X |

### Git Commit
`[hash]` — [message]
```

**文档债务控制**：每 5 个 Session 做一次全量审查（格式统一、去重、清理过时内容）。

### 纪律 2：双基线交叉校验（防止腐化）

两份基线，定位不同：

| 基线 | 文件 | 定位 | 更新时机 |
|------|------|------|---------|
| **设计基线** | `docs/baselines/design-baseline.md` | 实现驱动：我们造了什么 | Phase 结束后，对照代码更新 |
| **规划基线** | `docs/planning/plan-baseline.md` | 设计驱动：我们要造什么 | 讨论设计后更新 |

**交叉校验流程**（每个 Phase 结束时）：

```
Phase 完成
  ↓
更新设计基线（对齐实际实现）
  ↓
设计基线 ×× 规划基线交叉校验
  ↓
发现偏差 → 决策：修正规划 or 补齐实现
  ↓
两份基线保持一致
```

**交叉校验清单**：
1. API 端点、数据模型是否与代码一致？
2. Phase 状态是否反映实际完成情况？
3. 两份基线之间是否有矛盾？
4. 是否有残留的版本批注需要清理？

### 纪律 3：验收测试驱动（连接代码与产品）

**核心原则**：单元测试全过 ≠ 产品可用。
（Forge 教训：156 个单元测试全过，但 workspace 工具 WebSocket 路径可用、REST 路径不通。）

**验收测试生命周期**：

```
场景先行 → 编码实现 → 交叉验证 → 运行时执行 → 数据校准 → 更新文档
  ↑                                                        |
  └────────────── 下一个 Phase 时继承并扩展 ────────────────┘
```

**关键步骤**：

1. **场景先行**：每个 Phase **编码前**先写验收场景标题 + 关键预期
   （Forge 教训：验收测试写完代码后才补，Session 13 交叉验证发现 **16 个错误**）

2. **编码后补充**：填充 TC 操作细节，对照代码做交叉验证

3. **运行时执行**：可自动化的用例用脚本执行，UI 用例标注手动

4. **数据校准**：执行后将实际值回填文档（通过率、接口数等）

**验收测试格式**：
```markdown
### TC-N：[场景名]

> [简要描述]

**操作**：
1. 执行 `curl ...`
2. ...

**预期**：
- [ ] 返回 200
- [ ] body 包含 ...
```

### 纪律 4：防腐规则（持续改进）

**系统性 Bug 全局排查**
发现一个 Bug 属于某类系统性问题时，**立即全局排查同类问题，一次性修复**。
Forge 教训：枚举序列化 Bug 出现了 3 次（BUG-008/BUG-017/BUG-N），因为每次只修复当前文件。

```bash
# 发现枚举序列化问题时立即执行
grep -r "enum class" --include="*.kt"
# 对所有枚举统一加 @JsonValue
```

**反馈循环速度**
优先选择快速反馈的验证方式：

```
修改代码
  → 单元测试（~30s）✅
  → 类型检查 npm run build（~10s）✅
  → Docker 重建（~3-5min）⚠️ 仅用于最终集成验证
```

Forge 教训：Session 15 "改一行 → Docker 重建 3 分钟 → 发现新问题 → 重复"，4 次 Docker 重建消耗了大量时间。
**建立本地开发模式（bootRun + dev），把 Bug 发现提前到 Docker 之前。**

**Git 提交粒度**
- 每个逻辑变更一个 commit（一个 Bug = 一个 commit，一个功能 = 一个 commit）
- 禁止单个 commit 包含 20+ 文件的大杂烩
- Commit message 使用语义化格式：`feat:` / `fix:` / `docs:` / `refactor:`

**同一功能多路径测试**
同一功能的不同入口（REST API / WebSocket / UI）必须各自测试。
Forge 教训：workspace 工具 WebSocket 可用但 REST API 路径未连通，是完全不同的代码路径。

## 六、经验编码管道

将踩坑经验固化为永久知识，防止下一个 Session 重蹈覆辙：

```
踩坑（Session 中发现）
  ↓
Logbook 经验沉淀（当次 Session 记录）
  ↓
跨 2+ Session 验证（确认是普遍规律，非偶发）
  ↓
编码到 CLAUDE.md 已知陷阱（成为所有 Session 的先验知识）
```

**例外**：用户明确要求记住的内容可跳过验证直接编码。

## 八、已知的 AI Coding 反模式

以下是 Forge 28 个 Session 验证的反模式，Cruise 应避免：

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 不写 CLAUDE.md / 写了不更新 | 每次 Session 从零开始，重复犯同样错误 | Session 结束时沉淀经验到 CLAUDE.md |
| 验收测试"后补" | 测试写的是错误的场景，无法发现真实 Bug | Phase 开始时先写场景标题 |
| 单元测试全过 = 完成 | 集成路径的 Bug 无法发现 | 同时执行 E2E 验收测试 |
| 发现 Bug 只修当前文件 | 同类 Bug 下次再踩 | 立即全局排查同类问题 |
| 每次验证都 Docker 重建 | 3-5min 反馈循环，严重拖慢开发 | 建立本地开发模式 |
| 大杂烩 commit | 无法 bisect，回滚困难 | 每个逻辑变更一个 commit |
| Phase 编号反复变更 | Session 13 的 69 处重命名消耗整个 Session | 里程碑命名从一开始就稳定 |
| 并行 Agent 评估结果不验证 | Explore Agent 可能误判（Forge Session 2 案例） | 关键结论手动 double-check |
| 上下文窗口填满再工作 | 模型性能下降 20-30% | 75% 时主动 /compact |
| 文档债务积累不清理 | Session 13 专门花一个 Session 做文档重构 | 每 5 Session 做一次文档审查 |
