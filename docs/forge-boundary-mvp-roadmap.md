# Forge 目标边界、MVP 与演进路线

> 分支: `feature/architecture-refactor`
> 日期: 2026-03-12
> 目的: 在保留 Forge 长期目标的前提下，采用更收敛的产品与架构落地路径。

---

## 一、核心判断

Forge 的长期目标不应改变：它仍然应当是一个面向研发交付场景的 AI 平台。

但在落地路径上，Forge 不应一开始就实现完整的平台化能力集合，而应优先收敛为一个高频、稳定、可持续使用的交付闭环。

因此，当前阶段的核心策略是：

1. **保留 Forge 的目标**：AI 驱动的软件交付平台
2. **学习 Linear 的收敛方式**：先把一个核心工作流做到极强
3. **延后平台泛化**：MCP、Skill、Memory、治理能力按业务需要逐步长出

---

## 二、Forge 的目标边界

### 2.1 Forge 要解决什么问题

Forge 聚焦的是研发交付过程中的三类问题：

1. **执行问题**：任务从输入到产出依赖大量重复劳动，交付链路长、切换多、效率不稳定
2. **协同问题**：个人、团队、评审人之间的信息传递断裂，交付上下文容易丢失
3. **治理问题**：规范、基线、经验、工具分散，AI 产出质量难以稳定复现

### 2.2 Forge 不优先解决什么问题

当前阶段，Forge 不应扩展为以下产品：

- 通用项目管理平台
- 通用知识库平台
- 通用低代码平台
- 通用 IM 协作工具
- 面向所有岗位的企业 AI 门户

这些方向都可能成为未来连接能力的一部分，但不应成为当前产品边界本身。

### 2.3 Forge 当前应聚焦的业务边界

Forge 当前的业务边界应收敛为三层：

#### A. 交付执行层

这是最核心的产品层，负责完成一次真实交付任务的闭环：

`任务输入 -> 上下文装配 -> Agent 执行 -> 产物生成 -> 人工确认 -> 结果沉淀`

#### B. 团队治理层

在单次交付闭环稳定后，逐步增加团队级能力：

- 交付记录
- 审批与确认
- 规范与基线
- 可复用经验
- 团队级工作方式统一

#### C. 工程连接层

这是平台的底层接入层，用于连接外部工程系统：

- Git 仓库
- CI/CD
- 数据库与 schema
- 服务图谱
- 日志、指标、链路
- 制品与部署记录

当前阶段，工程连接层只需要覆盖支撑核心闭环所需的最小连接，不追求完整企业中台能力。

## 三、产品北极星

Forge 的北极星不应定义为“接入多少 MCP 服务”或“支持多少 Skill”，而应定义为：

**让研发人员以最少上下文切换完成一次高质量交付。**

围绕这个北极星，当前阶段最重要的产品指标应是：

1. 单任务完成时间是否下降
2. 首次交付通过率是否提升
3. 人工返工次数是否减少
4. 同一类任务的交付质量是否趋于稳定

---

## 四、MVP 定义

### 4.1 MVP 目标

MVP 不是“最小平台”，而是“最小可持续使用的交付闭环”。

MVP 只回答一个问题：

**用户能否在 Forge 中完成一次从任务输入到产物确认的完整过程，并愿意再次使用。**

### 4.2 MVP 用户

当前阶段只服务一类核心用户：

- **研发工程师 / 技术负责人**

不优先面向：

- 高层管理者
- 大规模非技术协作人群
- 复杂跨部门审批链路

### 4.3 MVP 核心场景

MVP 只保留以下高频场景：

1. 用户输入一个开发/修改任务
2. 系统装配与该仓库相关的上下文
3. Agent 生成实施计划并执行
4. 产出代码、文档或配置修改
5. 系统展示过程、结果与待确认项
6. 用户确认后沉淀结果

### 4.4 MVP 必要能力

#### 前端

- Web IDE
- 文件树与文件预览
- 对话面板
- 执行过程日志
- 结果确认面板

#### Agent

- 单一 SuperAgent
- 基于任务的轻量角色切换
- 基础计划能力
- 基础执行与总结能力

#### 上下文

- 仓库文件上下文
- 项目级规则文件
- 当前任务相关文件选择

#### 连接能力

- Git 工作区读写
- 基础命令执行
- 最少量知识检索

### 4.5 MVP 明确不做

- 多 Agent 编排
- 完整企业级审批流
- 完整多租户权限体系
- 全量 MCP 服务矩阵
- 复杂评估体系
- 高级经营分析看板
- 广义项目管理功能

## 五、架构收敛原则

### 5.1 先产品闭环，后平台抽象

凡是不能直接提升“任务输入到产物确认”闭环质量的能力，都不应优先平台化。

### 5.2 先单 Agent，后多角色体系

先把单一 SuperAgent 做稳定，再逐步把规划、设计、开发、测试等角色能力结构化，而不是一开始设计复杂 Agent 协作网络。

### 5.3 先最小连接，后完整生态

MCP 或外部工具接入必须围绕真实闭环需要增长，避免为平台完整性而接入。

### 5.4 先可用性，后完美治理

HITL、Baseline、Memory、Evaluation 都有价值，但只有在闭环可持续使用后，它们才值得持续投入。

### 5.5 先行为数据，后平台指标

优先记录真实任务执行数据、确认次数、返工原因、失败环节；不要过早做宏观平台大盘。

## 六、演进路线

### 6.1 v1：AI 开发工作台

#### 目标

把 Forge 收敛成一个可反复使用的 AI 开发工作台。

#### 交付重点

- 单任务闭环
- Web IDE 基础体验
- 过程透明
- 文件修改与结果确认
- 基础上下文装配

#### 成功标志

- 用户能稳定完成单仓库任务
- Agent 输出结果可审阅、可确认、可复用
- 同类任务重复使用意愿明确

### 6.2 v2：团队交付中枢

#### 目标

从“个人可用”扩展为“团队协同可用”。

#### 增长能力

- 团队共享记忆
- 交付记录归档
- 基线和模板统一
- 轻量审批与 Review
- 更明确的 Skill/Profile 路由

#### 成功标志

- 同一团队内的交付方式开始标准化
- 多人可共享上下文和最佳实践
- 文档、规范、经验不再散落在仓库外部

### 6.3 v3：企业工程控制面

#### 目标

把 Forge 提升为企业研发流程的 AI 编排与治理入口。

#### 增长能力

- 多项目治理
- 多系统统一连接
- 可观测性与制品追踪
- 质量门禁与合规规则
- 更成熟的 MCP 生态

#### 成功标志

- Forge 成为研发流程的统一入口之一
- 工程知识、流程、工具连接形成复用网络
- 平台治理能力开始替代人工流程协调

---

## 七、推荐的项目拆分方式

为避免后续再次失焦，建议把后续工作拆为三个持续主题：

### 7.1 主题一：闭环产品化

只关注用户是否能顺畅完成一次交付任务。

### 7.2 主题二：治理资产化

把规范、模板、经验、确认机制逐步变成结构化资产。

### 7.3 主题三：连接平台化

围绕真实业务需求，逐步扩展 Git、CI、数据库、观测和制品能力。
