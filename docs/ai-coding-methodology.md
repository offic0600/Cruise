# AI Coding 项目开发方法论

> 基于 Forge 平台项目实战经验提炼（28 个 Session，5 天，50K+ 行代码，91% 验收通过率）

---

## 一句话核心

> **用文档对抗 AI 遗忘，用验收测试对抗质量腐化，用双基线对抗设计偏移。**

AI Coding 的本质矛盾：Claude Code 的上下文窗口是有限的，但项目的复杂度是无限增长的。
解法：在 AI 之外建立**外部记忆系统**，用纪律弥补 AI 的遗忘。

---

## 第一层：三大基础设施

这是 AI Coding 项目的地基，必须在项目第一天就建立，并贯穿始终。

### 1.1 CLAUDE.md — AI 的"项目大脑"

**作用**：每个新 Session 启动时，Claude 自动读取 CLAUDE.md，恢复项目知识。

**必须包含的内容**：

| 章节 | 内容 | 示例 |
|------|------|------|
| 交互偏好 | 语言、风格、行动优先还是确认优先 | "使用中文回复，行动优先，不做冗余解释" |
| 架构快照 | 模块划分、技术栈、关键依赖 | "后端 Spring Boot 3，前端 React 18，PostgreSQL" |
| 已知陷阱 | 踩过的坑，防止重复踩 | "Kotlin 枚举必须用 @JsonValue 注解，否则序列化为 ordinal" |
| 开发纪律 | 强制规则，AI 必须遵守 | "任何变更都要先跑 `./gradlew test`" |
| 构建命令 | 常用命令速查 | `./gradlew bootRun`、`npm run dev`、`docker compose up` |

**演进规律**：从 ~50 行起步，随项目推进持续增长，每发现一个"经验"就补充进来。67 行 → 181 行是健康的演进轨迹。

> ⚠️ **反模式**：CLAUDE.md 只有 10-20 行基础配置 = 每个新 Session 都要重新解释项目背景，效率极低。

---

### 1.2 Dev Logbook — 项目的"时间记忆"

**作用**：记录每个 Session 发生了什么、为什么这样做、踩了什么坑。对抗项目遗忘的核心武器。

**Session 结构（每次固定格式）**：

```markdown
## Session N — [日期] [标题]

### 目标
本次 Session 要完成什么

### 实施记录
- 做了什么（关键决策，不是流水账）
- 遇到什么问题，如何解决

### 文件变更表
| 文件 | 操作 | 说明 |
|------|------|------|
| src/xxx.java | 新增 | 原因 |

### Bug 记录
- BUG-N: [现象] → [根因] → [修复]

### 经验沉淀
- 本次 Session 学到了什么，值得编码到 CLAUDE.md 的内容

### 统计快照
文件数 / 代码行数 / 测试数 / 通过率 / Bug 数
```

**关键原则**：
- 每个 Session 结束前**必须**更新 Logbook，否则下个 Session 记忆断链
- 重点写**决策背后的原因**，不是代码本身（代码在 git 里，原因只在 Logbook 里）
- 每 5 个 Session 做一次**文档精简**，避免 Logbook 变成负担

---

### 1.3 双基线系统 — 项目的"空间记忆"

两份文档独立演进，Phase 结束后交叉校验：

```
规划基线（planning-baseline.md）  →  我们要造什么  →  设计驱动
设计基线（design-baseline.md）    →  我们造了什么  →  实现驱动
```

**规划基线**：描述产品愿景、功能范围、技术选型、Phase 路线图。
**设计基线**：描述当前已实现的架构、API 契约、UI 交互、关键数据模型。

**为什么需要两份**：规划基线防止"忘了要做什么"，设计基线防止"忘了已经做了什么"。两者的偏差就是 Gap，是下一个 Phase 的规划输入。

> **实战教训**：Phase 1.5 Docker 部署成功后，没有设计基线，下一个 Phase 的 AI 无法判断"当前 UI 布局是已验证的设计还是可以随意修改的"。设计基线就是"已验证设计的冻结快照"。

## 第二层：Phase 迭代开发节奏

### 2.1 Phase 结构

每个 Phase 是一个**功能聚合单元**，不是时间盒：

```
Phase 0（骨架）→ Phase 1（核心功能）→ Phase 2（增强）→ Phase N（...)
```

**Phase 内部结构**：

```
Phase 开始
  └─ 更新规划基线（这个 Phase 要做什么）
  └─ 编写验收场景（先写，不是后补！）
      │
      ├─ Session 1: 核心功能
      ├─ Session 2: 增强 + Bug 修复
      ├─ Session N: ...
      │
  └─ 执行验收测试
  └─ Bug 修复直到通过率 ≥ 目标（建议 ≥ 90%）
  └─ 更新设计基线（这个 Phase 造了什么）
  └─ 提炼经验 → 更新 CLAUDE.md
Phase 结束
```

**Phase 大小建议**：
- 太小（<3 个 Session）：启动成本太高，还没建立 momentum 就结束了
- 太大（>8 个 Session）：中途难以度量进度，Bug 积累太多
- **建议 4-6 个 Session 为一个 Phase**

---

### 2.2 Session 作为最小工作单元（微型 PDCA）

每个 Session = 一次完整的 Claude Code 对话，约 2-3 小时。

```
Plan（目标声明）
  → Do（实施：代码/文档/测试）
    → Check（跑测试，人工验证）
      → Act（经验沉淀，更新 CLAUDE.md / Logbook）
        → Git Commit & Push
```

**Session 启动仪式**（新 Session 开始时固定做）：
1. 向 Claude 声明："请先读 CLAUDE.md 和 Logbook 最近 N 个 Session"
2. 确认 Claude 理解当前项目状态（问："目前的架构是什么？上次 Session 做到哪了？"）
3. 声明本次 Session 目标

**Session 结束仪式**（Session 结束前固定做）：
1. 更新 Logbook（文件变更 + Bug + 经验沉淀 + 统计快照）
2. 提炼 1-3 条经验到 CLAUDE.md
3. Git Commit（语义化消息）

## 第三层：验收测试驱动（ATDD）

### 3.1 核心原则：先场景，后代码

```
❌ 错误顺序：写代码 → 写验收测试 → 执行
✅ 正确顺序：写验收场景（至少标题 + 预期）→ 写代码 → 填充测试细节 → 执行
```

**为什么要先写场景**：
案例：Session 12 代码完成后写验收测试（89 用例），Session 13 交叉验证发现 **16 个测试本身写错了**。先写场景可以把"需求理解"的校准提前到编码前，而不是在代码完成后才发现理解偏差。

### 3.2 验收测试文档结构

```markdown
## 场景 N：[功能名称]

| 用例 | 操作 | 预期结果 | 实际结果 | 状态 |
|------|------|---------|---------|------|
| N.1 | [具体操作] | [具体预期] | - | ⬜ |
| N.2 | ... | ... | - | ⬜ |
```

**状态标记**：⬜ 待测 / ✅ 通过 / ❌ 失败 / ⚠️ 部分通过

**执行纪律**：
- 验收测试执行时，AI 逐条执行，人工记录实际结果
- 失败的用例立即记录为 Bug（带编号）
- Phase 结束时统计通过率，作为 Phase 质量门禁

### 3.3 质量门禁建议

| 指标 | 最低要求 | 建议目标 |
|------|---------|---------|
| 验收通过率 | ≥ 80% | ≥ 90% |
| Bug 修复率 | ≥ 85% | ≥ 95% |
| 单元测试通过 | 100% | 100% |
| P0 Bug 数 | 0 | 0 |

## 第四层：Bug 管理闭环

### 4.1 Bug 记录规范

每个 Bug 必须有：

```markdown
### BUG-N：[现象标题]
- **等级**：P0（阻断）/ P1（严重）/ P2（一般）/ P3（轻微）
- **现象**：用户看到的表现
- **根因**：技术层面的原因
- **修复**：如何修复
- **防止复发**：编码到 CLAUDE.md 的陷阱描述
```

### 4.2 系统性 Bug 规则

**当发现一个 Bug 时，立刻判断：这是个例还是系统性问题？**

- **个例**：修复这一处即可
- **系统性**：立刻做全局排查，一次修完所有同类

```bash
# 发现枚举序列化问题后，立刻全局排查
grep -r "enum class" --include="*.kt"
```

> **实战教训**：枚举序列化问题出现两次（BUG-008 和 BUG-017），Session 16 意识到"应全局排查"但未执行，导致第二次踩同一个坑。系统性 Bug 必须一次清零。

### 4.3 根因分析价值

Bug 的根因分析不只是为了修复这个 Bug，而是为了：
1. 发现**同类 Bug**（同一根因的不同表现）
2. 识别**设计缺陷**（Bug 反复出现的模块，通常有架构问题）
3. 补充 **CLAUDE.md 陷阱列表**（防止 AI 在新 Session 重蹈覆辙）

## 第五层：AI 工具使用最佳实践

### 5.1 Plan Mode 使用时机

**必须先 Plan 再 Execute 的情况**：
- 涉及 5+ 文件修改
- 新建模块或重构架构
- 不确定实现路径的功能

**可以直接 Execute 的情况**：
- 明确的 Bug 修复（已定位根因）
- 单文件的功能增强
- 文档更新

### 5.2 并行 Agent 使用

**适合并行的任务**（互不依赖）：
- 前端功能 + 后端 API 同时开发
- 多个独立模块同时创建
- 验收测试并行执行（不同场景分组）
- 文档并行更新（Logbook + 基线 + 验收测试）

**不适合并行的任务**：
- 有依赖关系的功能（A 要调用 B 的接口）
- 共享状态的修改（同一个配置文件）

### 5.3 反馈循环速度（关键！）

**反馈循环越短，效率越高**：

| 层次 | 工具 | 耗时 | 使用频率 |
|------|------|------|---------|
| 单元测试 | `./gradlew test` | ~30s | 每次修改后 |
| 类型检查 | `npm run build` | ~10s | 每次修改后 |
| 本地集成 | `./gradlew bootRun` + `npm run dev` | ~30s | 每个功能完成 |
| Docker 集成 | `docker compose up --build` | 3-5min | Phase 验收前 |
| 全量验收 | 手动 + 自动 | 1-2h | Phase 结束 |

**核心原则**：**绝大多数 Bug 应在 Docker 之前发现**。不要让 Docker 构建成为主要的 Bug 发现机制。

> **实战数据**：Session 15 用 4 次 Docker 重建发现 Bug，每次 3-5 分钟 = 20 分钟的等待。如果用本地运行模式，同样的 Bug 可以在 30 秒内发现。

### 5.4 Git 提交纪律

- **每个逻辑变更一个 Commit**（便于 bisect 和回滚）
- 使用语义化前缀：`feat:` / `fix:` / `docs:` / `refactor:` / `test:`
- Commit 消息写"为什么"不只是"做了什么"

**反模式**：一个 Commit 包含 200+ 文件 / 30K+ 行 → 无法有效 review，无法 bisect

## 第六层：上下文管理（对抗 AI 遗忘）

### 6.1 四维记忆体系

| 记忆维度 | 载体 | 回答的问题 |
|---------|------|-----------|
| 时间维度 | Dev Logbook | 过去发生了什么？为什么这样做？ |
| 空间维度 | 设计基线 + 规划基线 | 系统当前的全貌是什么？ |
| 质量维度 | 验收测试 + Bug 列表 | 什么能工作？什么不能？ |
| 知识维度 | CLAUDE.md + 经验沉淀 | 我们学到了什么？ |

### 6.2 Context 管理

Claude Code 上下文窗口约 200K token，分四个区间：

| 状态 | 使用量 | 行动 |
|------|--------|------|
| 绿色 | <50% | 正常工作 |
| 黄色 | 50-75% | 考虑 /compact |
| 橙色 | 75-90% | 执行 /compact，精简非必要内容 |
| 红色 | >90% | 结束当前 Session，开新 Session |

**Session 切换时**：新 Session 启动时主动声明"请先读 CLAUDE.md 和最近的 Logbook"，不要假设 Claude 记得上次的内容。

### 6.3 CLAUDE.md 演进节奏

```
Session 1: 基础配置（架构 + 命令 + 偏好）
Session 3-5: 加入已知陷阱（从 Bug 提炼）
Session 8-10: 加入开发纪律（从流程提炼）
Phase 结束: 全面审查，精简过时内容
```

## 第七层：持续改进机制

### 7.1 "痛点即能力"原则

> 我们自己踩的坑，就是工具/流程应该帮我们自动规避的坑。

**实践方法**：
- 每次 Phase 结束，做"痛点回顾"（什么最浪费时间？什么最容易出错？）
- 把能自动化的痛点变成脚本或检查规则
- 把不能自动化的痛点编码到 CLAUDE.md 作为 AI 指令

**案例**：Docker 部署 90 分钟 → 发现 7 类可提前检测的问题 → 编写 pre-flight check 脚本 → 下次部署 5 分钟。

### 7.2 Phase 结束复盘（固定模板）

```markdown
## Phase N 复盘

### 交付数据
- 计划功能 N 个，完成 N 个（完成率）
- 验收通过率：N%
- Bug 发现 N 个，修复 N 个

### 最有价值的决策
1. ...

### 最大的教训
1. ...

### 下个 Phase 改进点
1. ...

### 编码到 CLAUDE.md 的新内容
1. ...
```

### 7.3 文档债务管理

文档是资产，但也会成为负担。**每 5 个 Session 安排一次文档精简**：
- 合并重复内容
- 删除已过时的章节
- 将长描述压缩为要点
- 可以让 AI 执行，但人工审查精简结果

## 快速上手：新项目第一周

### Day 1（Session 1）：建立地基
- [ ] 创建 CLAUDE.md（基础版：架构 + 命令 + 偏好）
- [ ] 创建 dev-logbook.md（Session 1 模板）
- [ ] 创建 planning-baseline.md（Phase 0 范围）
- [ ] 用 Plan Mode 生成项目骨架
- [ ] 第一次 Git Commit

### Day 2-3（Session 2-4）：建立核心功能
- [ ] 每个 Session 开始前宣读目标
- [ ] 每个 Session 结束前更新 Logbook
- [ ] Phase 1 验收场景**先于**编码编写
- [ ] 本地测试通过后再考虑集成测试

### Day 4（Session 5-6）：质量验收
- [ ] 执行 Phase 1 验收测试
- [ ] Bug 全部记录、分级
- [ ] 修复直到通过率 ≥ 90%
- [ ] 更新设计基线（Phase 1 造了什么）
- [ ] Phase 1 复盘，提炼 CLAUDE.md

### Day 5（Session 7+）：下一个 Phase
- [ ] 检查规划基线和设计基线的 Gap
- [ ] 确定 Phase 2 范围（先写验收场景）
- [ ] 循环...
